💡 Accepts .pem, .crt, .cer, .key. For a single bundle file with multiple blocks, upload to Certificate — the parser auto-splits cert/key/chain. .p12 / .pfx not supported; convert with openssl pkcs12 -in file.p12 -out file.pem -nodes first.
Staging certs are untrusted but rate-limits are loose — use to validate setup before going live.
ℹ️ HTTP-01 challenge requires this server to be reachable at http://<domain>/.well-known/acme-challenge/* from the internet. The main HTTP server already serves these.

Response payloads
POST /oauth/token — after the authorization-code exchange
{
  "access_token": "<JWT>",
  "id_token":     "<JWT>",
  "token_type":   "Bearer",
  "expires_in":   86400,
  "scope":        "openid profile email",
  "refresh_token": "<opaque>"
}

access_token and id_token carry the same RS256 JWT — verify via the JWKS endpoint above.

JWT claims — decoded payload of access_token / id_token
{
  "iss":   "<JWT_ISSUER>",
  "sub":   "<uuid-v5 derived from midleman_uid>",
  "aud":   "<oauth_client_id>",
  "iat":   1730000000,
  "exp":   1730086400,
  "midleman_uid":        42,
  "preferred_username":  "jdoe",

  // included when "email" scope is granted
  "email":          "jdoe@example.com",
  "email_verified": true,

  // included when "profile" scope is granted
  "name": "John Doe",

  // included when the authorize request had a nonce
  "nonce": "<client-supplied>"
}
GET /oauth/userinfo — with Authorization: Bearer <access_token>
{
  "sub":                "<uuid>",
  "preferred_username": "jdoe",
  "email":              "jdoe@example.com",
  "email_verified":     true,
  "name":               "John Doe",
  "midleman_uid":       42
}
Refresh: POST /oauth/token — grant_type=refresh_token
{
  "access_token":  "<new JWT>",
  "id_token":      "<new JWT>",
  "token_type":    "Bearer",
  "expires_in":    86400,
  "scope":         "openid profile email",
  "refresh_token": "<rotated opaque>"
}

Refresh tokens are single-use and rotate on every call. The previous token is invalidated.

Errors — RFC 6749 §5.2 shape
{
  "error":             "invalid_grant",
  "error_description": "Authorization code expired or already used"
}

Email (SMTP)

Configure the SMTP server used to deliver outgoing email from Midleman — including admin invites.

Server

Address of the SMTP server that will deliver the mail. Usually provided by your email provider (Gmail, Outlook 365, SendGrid, Mailgun, etc.) or by your organisation's mail server.

Sending credentials

Account used by Midleman to authenticate against the SMTP server. Without it the server will usually reject the message (unless it accepts anonymous relay from your network). For Gmail/Outlook it is typically an app password; for SendGrid/Mailgun it's the service username and API key. Leave both empty if your server does not require authentication.

Leave empty to keep the current password.

Sender identity

What appears in the From: header of outgoing messages. Recipients will see this address as the origin of the message and will reply to it. It must be an address the SMTP server is authorised to send from — otherwise messages may be rejected or flagged as spam.
